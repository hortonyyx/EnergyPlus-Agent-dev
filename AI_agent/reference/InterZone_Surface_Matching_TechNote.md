# EnergyPlus InterZone 面相交与匹配（Surface Intersection & Matching）— 技术参考

> 独立技术备忘。主题：把一个几何建筑模型中**相邻热区（及上下相邻楼层）之间、面不一一对应**的边界，确定性地切分并匹配成 EnergyPlus 要求的**一一对应**面对。供实现该模块的工程参考，不含具体实现选型决策。

---

## 1. 问题定义

EnergyPlus 把每个热区（Zone）建模为一组平面面（`BuildingSurface:Detailed`）。两个热区相邻、或上下两层相叠时，它们之间的**共享边界必须表示成一对面**（每个热区各一个），且这一对面：

- **几何全等**：共面、面积相等、法向相反；
- **互相引用**：各自 `Outside Boundary Condition = Surface`，`Outside Boundary Condition Object` 指向对方；
- **构造层互逆**：两面的 Construction 材料层栈互为反序（或共用同一 Construction，平凡满足）。

当相邻两侧的划分**不一致**时（典型：一个大房间上方对应下方三个小房间，或上下楼层隔墙不对齐），一侧的单个面会与另一侧多个面交叠。EnergyPlus **只接受一对一**，因此较大的面必须被**切分成多片**，每片**恰好匹配**对侧一个面。

这是一个计算几何操作：找出共面交叠的候选面对 → 求交叠多边形 → 在所有交叠伙伴诱导的断点并集处切分 → 为每片建立互逆边界条件。等价于 OpenStudio 的 `intersect` + `match` 两步。

---

## 2. EnergyPlus 的硬性要求（违反即 warning/severe/fatal）

| 要求 | 不满足的后果 |
|---|---|
| InterZone 面成对、互相引用且引用对象存在 | `…references an outside boundary surface that cannot be found` fatal |
| 配对两面面积相等（容差内） | 面积不匹配 severe / 热平衡异常 |
| 配对两面共面、法向相反 | 几何告警 / 传热错误 |
| 一片只被匹配一次（无重复引用） | 边界条件歧义 |
| 配对两面 Construction 层互为反序 | `GetSurfaceData: Construction … does not have the same materials in the reverse order …` fatal |
| 每条边长 ≥ 最小阈值（退化面） | 输入处理阶段崩溃（见 §4.1） |

---

## 3. 为什么用确定性几何算法（而非自然语言/LLM 推理）

切分本质是 **O(n×m) 的断点并集枚举 + 多边形布尔运算**，对数值精度敏感。用自然语言推理来做这一步脆弱且不可复现。

**实践观察到的失败模式**：当上下相邻两面在某轴上存在约 5 cm 的不一致，切分会切出约 0.1 m 量级的**退化窄条面**（如 4.90–4.95 这样的 5 cm × 3 m 碎片）。EnergyPlus 在输入处理阶段对这类退化面**直接崩溃**（core dump，且在写出 `.err` 之前即崩，故错误日志为空、难以定位）。这类问题靠确定性的断点吸附 + 退化面剔除才能根除。

结论：相交/匹配/切分**应是确定性代码**；它对标 OpenStudio 几十年工业级的 `intersect`/`match` 实现，不应重新发明成文本推理。

---

## 4. 算法骨架

输入 = 一组热区体块（每个有底面多边形 + z 范围 → 6 个或更多平面面）。
输出 = 切好、匹配好、设好互逆边界条件的面集合。

1. **候选面对检测**：对所有面两两检查，筛出**共面**（同一平面，法向夹角 ≈180°）且**投影多边形交叠**的面对。空间索引（如 R-tree / 按平面分桶）避免真 O(N²)。
2. **投影求交**：把候选两面投影到公共平面（转 2D），用多边形布尔运算求交叠多边形。
3. **切分**：把每个面在**所有交叠伙伴诱导的断边并集**处切分，使每个结果片**恰好交叠对侧一片**。
4. **匹配**：为每片设 `OBC = Surface` + `OBC Object = 对片`；校验面积相等、法向相反、共面。
5. **构造反序**：配对两面共用单一 Construction（推荐，平凡满足反序）或显式保证层栈互为反序。

## 5. 两条实现路线

### 5.1 自写（建在 shapely + 一个 IDF 数据模型库上）
- **shapely**：平面集合代数（多边形交/并/差、断点切分）——切分主力。注意面是 **3D 平面多边形**，需先投影到面所在平面做 2D 运算再抬回 3D。
- **IDF 数据模型库**（带 `area/normal/centroid` 几何 mixin、`surface→zone` 导航的那类）：替代裸文本/低层 IDF 操作，降低样板代码量。
- 两者**互补**：数据模型库出干净的 surface 多边形，shapely 做布尔运算；数据模型库**替代不了** shapely 的布尔运算。

### 5.2 OpenStudio SDK（intersect + match）
- `Surface::intersect` + `Surface::matchSurfaces`（或 "Match Surfaces" measure）成熟、久经考验。
- 代价 = 引入 OpenStudio SDK 依赖（较重），需评估部署 / license / 与现有 IDF 栈的衔接。

> 选型（能力覆盖 / 依赖重量 / license / 引入成本）由实现方权衡，本备忘只列两条路。

## 6. 关键工程注意

### 6.1 容差与退化面（最致命）
- 切分前**把断点吸附到统一坐标系/规范轴集**，把"上下两层同一道墙差 5 cm"这类不一致**先归一**，再切分；否则必生退化碎片。
- 切分后**剔除/吸收低于最小边长阈值的退化片**（建议阈值 ≥0.1 m，与 EnergyPlus 崩溃边界对齐）。
- 共面 / 法向相反 / 点在面上 / 多边形交叠 各判定的容差要一致。建议坐标先**量化到固定栅格**（如 50 mm）压浮点噪声。

### 6.2 schema 校验 ≠ 几何校验
IDF 数据模型库的 `validate()` 只查 **schema 合规**，**不验几何**。有覆盖洞或配对错的模型可以 schema 合法但几何错、仿真崩。**别指望 schema 校验抓几何缺陷。**

### 6.3 覆盖完整性 vs 配对合法性（两件事）
- **配对合法性** = 已声明的配对是否几何合法（§4 算法 + §7 校验覆盖这个）。
- **覆盖完整性** = 有没有"本该是内部边界、两侧却都被标成 Outdoors/Adiabatic、于是根本不进配对图"的洞。**逐对匹配器查不到这类洞，EnergyPlus 运行也不报错**（每个面单独合法）。
- 检测覆盖洞需 footprint 集合代数：相邻层 footprint 求并→求交得"该铺满的范围"，所有水平配对多边形求并得"实际铺到的范围"，求差 ≠ 空 = 洞。用 shapely。
- **若上游能保证输出是合法平面剖分（恰好铺满、无缝无叠），覆盖洞从源头不可能产生**，无需事后查。这属上游建模端的保证，与本模块互补。

### 6.4 数值鲁棒性
布尔运算在退化输入（共线、近重合点、零面积交叠）下易产生伪小片。投影到 2D 后用稳健的多边形库 + 统一容差 + 量化，可大幅减少伪片。

## 7. 装配后校验（验收）

匹配完成后做一遍确定性校验（一个正确的内核应让它退化成 sanity check）：每个 InterZone 面——
1. `OBC Object` 存在且是 Surface；
2. 互相引用（reciprocal）；
3. 被引用恰好一次；
4. 面积相等（容差内）；
5. 法向相反；
6. 共面（点到面距离容差内）；
7. 每条边长 ≥ 最小阈值。

任一不过即 fail-fast 报出可定位 issue，挡在 EnergyPlus 之前。

---

## 8. 竖向例外（提前规划，别后补）

"上下层共享同一 footprint、垂直升起"的简化假设会被以下情形打破，匹配器需支持：

- **退台（setback）**：上层 footprint 较小 → 下层天花一部分**合法地是 Outdoors（露出屋面）**、一部分是内部楼板。单个面需切成**内部匹配片 + 外部片**两部分。
- **中庭/通高（atrium）**：热区跨层无楼板 → 相邻层之间**没有**floor/ceiling 配对，是合法的"无伙伴"。匹配器不能把"找不到伙伴"一律当错。
- **斜屋面**：顶面非水平 → 共面/法向判定按非水平面处理。

建议主干用"逐层 2D 剖分 + 升起"，竖向现象作**显式例外标注**叠加，不退回纯 3D 推理。

## 9. 现成方案对照

- **OpenStudio**：`Surface::intersect` / `matchSurfaces`，工业级参考实现，可直接借或对照正确性。
- **shapely**：平面布尔运算事实标准。
- **R-tree（如 `rtree`/`STRtree`）**：候选面对检测的空间索引。
- 自动热分区 / BIM-to-BEM 工具链（如 OpenStudio 的 perimeter-core 划分）属**上游建模**范畴，与本模块（已给定热区后的面匹配）解耦——本模块吃任何粒度的热区体块都是同一算法。
