# 闭源 → 开源模型迁移判定准则

> 决策文档：回答「Claude Opus 上的 skill / prompt 调试到什么程度，才应该 pivot 到开源本地微调？上限一直上不去怎么办？」

---

## 零、核心原则

**闭源旗舰模型（Claude Opus）不是「起点」，而是「能力上限的探针」。**

它回答的是：**在最好的提示词 + 最强的通用智能下，这个任务到底能做到多好？**

- **上限高** → 开源微调是「把成本降下来」的工程问题，有 playbook
- **上限低** → 开源微调救不了，必须改任务定义或改输入表示

本文档的两个决策（Pivot / 不 Pivot）本质上是同一棵树的不同分支。

---

## 一、Pivot 阈值（必须双达标才切）

所有指标口径与 [plan.md](plan.md) §四 P0 的评测体系一致。

### 1.1 视觉层阈值（VLM 感知能力）

| 指标 | Pivot 阈值 | 当前估计 | 意义 |
|---|---|---|---|
| **zone 数匹配率** | ≥ **90%**（13 案例中至少 12 个完全对） | ~75%（6/8，未完样本未计）| 房间/走廊识别基本稳定 |
| **房间尺寸误差（中位数）** | ≤ **5%** | 未量化 | 尺寸链 OCR 可用 |
| **走廊 F1** | ≥ **0.85** | 未量化 | 宽白带识别稳定 |
| **特殊 zone（楼梯/WC）F1** | ≥ **0.80** | 未量化 | 符号识别可用 |

### 1.2 流水线阈值（下游可消费性）

| 指标 | Pivot 阈值 | 当前基线（[claude.md §3.1.2](claude.md)）|
|---|---|---|
| claude_ep.md 合规率 | ≥ 95% | 92%（12/13）|
| YAML 生成率 | ≥ 85% | 62%（8/13）|
| IDF 生成率 | ≥ 75% | 54%（7/13）|
| **EP 仿真完成率** | ≥ **50%** | **0%**（0/13）|

### 1.3 判定规则（四象限）

| 视觉层 | 流水线 | 含义 | 行动 |
|---|---|---|---|
| ≥ 阈值 | ≥ 阈值 | 任务有解，只是成本贵 | ✅ **可以 Pivot**。Opus trace 蒸馏 SFT 开源 VLM |
| ≥ 阈值 | < 阈值 | 视觉不是瓶颈，下游才是 | ⛔ **别 Pivot 视觉模型**。先解决 Schedule/Lights/HVAC 漏调 |
| 接近阈值但未达 | — | 还在爬坡，未触顶 | ⏸ **继续调 prompt**，别切模型 |
| 明显低于阈值且已收敛 | — | 任务触顶低于阈值 | ⚠️ **进入下一节「低上限应对」** |

**当前状态自查**：视觉层只有 zone 匹配率有粗略估计（~75%），其余三项未量化；流水线 EP 完成率 0%。**所以现在还没到判定时机**——必须先完成 [plan.md](plan.md) P0 评测基线，再看 Opus 摸到哪里。

## 二、阈值为什么定在这些数字

### 2.1 SFT 开源模型的一般规律
SFT 能把 7B–32B 开源 VLM 拉到「原始闭源模型能力的 80–90%」，拉不到同水平。所以：
- 闭源 90% → 开源 SFT 后约 75–80%，勉强可用
- 闭源 75% → 开源 SFT 后约 60–65%，不如不 Pivot

### 2.2 EnergyPlus 的二值容错
几何对就能仿真，错一个墙就 Fatal。视觉层 < 90% 意味着 ≥ 1/10 案例会在仿真阶段硬塌，流水线通过率不可能过半。

### 2.3 数据成本锚点
要把开源 VLM SFT 到闭源 90% 的水平，通常需要 **500–2000 条高质量 (图, 结构化输出) 对**。现有 13 个案例远远不够。**Pivot 前必须先用闭源模型批量生成训练数据**——这是为什么视觉层阈值要定在 ≥ 90%：只有高质量 Opus 产出才配当训练标签。

## 三、如果闭源上限一直上不去怎么办

### 3.1 先诊断「触顶原因」

```
Opus 视觉得分低且不再涨
├── 是不是图像本身信息不够？
│   → 放大、增强对比度、补充视图
├── 是不是 prompt 让 Opus 一次做太多？
│   → 再拆 CoT（plan.md §2.2 路线继续推）
├── 是不是输出格式太宽松让 Opus "偷懒"？
│   → 用 Pydantic / JSON schema 硬约束每一步
├── 是不是任务本身欠定义（比如房间功能靠猜）？
│   → 加 ground truth 先验（提供楼层功能分区表）
└── 以上都做了还不行 = 任务对通用 VLM 过难 → 进 3.2
```

### 3.2 真触顶后的 4 条退路（按 ROI 排序）

| 退路 | 思路 | 工程成本 | 适用情形 |
|---|---|---|---|
| **A. 前置视觉小模型** | 回到 [plan.md 方案 2](plan.md) 的最小模块（PaddleOCR + cv2 走廊）| 低 | 问题集中在数字识别 / 宽白带区分 |
| **B. 缩减任务（DSL 替代自由形式）** | 不让模型输出 `IntakeOutput`，改输出受限 DSL（如「房间格子语言」），程序翻译成 IDF | 中 | 问题集中在模型跑偏格式 |
| **C. Human-in-the-Loop** | LLM 出草稿 → 人工 10 秒级修正 → 入 IDF | 低 | 研究阶段够用，产品化不行 |
| **D. 闭源模型蒸馏专用小模型** | 即使闭源只有 75%，拿其 trace SFT 专用 VLM，可过拟合特定建筑风格 | 高 | 建筑风格集中、样本可扩到 500+ |

### 3.3 反直觉建议

如果闭源封顶 ~75% 且你能弄到 **≥ 500 对**中文办公建筑图纸的 (图, 结构化分区)，**直接走 D 路径反而可能比继续调 prompt 划算**：领域专用 SFT 模型在封闭任务分布上超过通用 VLM 是常态（类似医学影像 SFT 超过 GPT-4V）。

**但这条路的数据成本在 SFT 本身之前**——500 对数据，按 Opus 当前 75% 准确率、每对人工 5 分钟修正，等于**40+ 小时标注工作**，别低估。

### 3.4 候选：前置矢量化小模型（退路 A 的可选强化版）

**FloorplanVLM**（arXiv 2602.06507，Beike，2026-02）是退路 A 的潜在强化项——把 raster 平面图端到端吐成结构化 JSON（walls + openings + rooms + 邻接），形态上正好能替代现有 intake 中"LLM 看图 + 画 ASCII 平面图 + 编邻接矩阵"的整段工作。

**2026-04-21 评估结论：暂不纳入路径**，原因：

| 维度 | 现状 |
|---|---|
| 模型权重 | 未开源（arXiv / GitHub / HF 全库零命中） |
| 训练代码 & 数据集 | 未开源（Floorplan-2M / HQ-300K 均为贝壳内部） |
| FPBench-2K 基准 | 论文承诺发布，尚未见托管 |
| 基座 | Qwen2.5-VL-3B（开源，但 32×H200 自训成本不现实） |

即使未来开源，**仍有 3 个结构性不匹配**使其不能直接用：

1. **领域错位**：训练集是住宅室内设计（Manhattan 63.7%，语义标签 bedroom/kitchen/bath），与办公楼 SmallOffice（走廊/楼梯/WC/电梯）不重合；
2. **输出缺关键维度**：坐标归一化到 [0,1024] 像素 → **无米制尺度**，而 `create_zone` 需要真实世界坐标；也不输出楼层堆叠（`floors`/`zones_per_floor`），仍需外挂 OCR 读尺寸链反算比例 + 楼层逻辑；
3. **不解决当前瓶颈**：[CLAUDE.md §7.2](CLAUDE.md) 已结论视觉不是瓶颈，真瓶颈在长链路 tool-call 稳定性 + Schedule/Lights/HVAC 漏调——即便前置一个完美矢量化器，EP 完成率不会因此从 0 跳到 50%。

**触发再评估的条件**（任一满足）：
- 官方放出权重 + FPBench-2K，且 P1 在视觉层触顶仍低于 §1.1 阈值；
- Pivot 判定落入第四象限（触顶低于阈值且已收敛），此时走 3.2 退路 A，把 FloorplanVLM 作为 A 的增强方案与 PaddleOCR 并列评估。

**轻量跟踪动作**：订阅 arxiv 2602.06507 v2 更新；若 FPBench-2K 先于模型释出，可仅拿来做 [plan.md](plan.md) 评测脚本的外部 sanity check。

## 四、Pivot 后的技术路径（当判定通过时）

### 4.1 数据准备（Week 8–10）
1. 用 Opus 对更多建筑图纸（目标 ≥ 500 张）刷出 `(image, claude_ep.md, IntakeOutput)` 三元组
2. 人工抽检 10% 修错 → 形成 `sft_dataset_v1.jsonl`
3. 单独准备 `eval_holdout.jsonl`（≥ 50 条，人工 100% 校对）

### 4.2 模型选型（Week 11）
| 候选 | 显存 | tool-calling | 建议用途 |
|---|---|---|---|
| **Qwen2.5-VL-7B** | 单卡 24G 可跑 | ⭐⭐⭐⭐ | 首选试点 |
| **Qwen2.5-VL-32B** | 2 卡 80G | ⭐⭐⭐⭐ | 如果 7B 不够 |
| **InternVL3-8B** | 单卡 24G | ⭐⭐⭐ | 备选 |
| **MiniCPM-V 2.6** | 单卡 16G | ⭐⭐ | 轻量对照 |

### 4.3 训练方式（Week 12–14）
1. **LoRA SFT**（首选）：仅训 rank=16 的 LoRA，1 张 A100 / H100，约 6–12 小时
2. 评估：在 holdout 上复用 [plan.md](plan.md) P0 指标，**必须超过 Opus 的 80%**（即领域内不能比闭源差太远）
3. 若 LoRA 不够：全参 SFT（显存翻倍、时间翻 3 倍）

### 4.4 部署（Week 15+）
1. vLLM 启 OpenAI 兼容 API
2. [../src/configs/llm.yaml](../src/configs/llm.yaml) 改 `provider: openai`, `base_url: http://localhost:8000/v1`
3. 跑完整 13 案例回归，与 Opus 基线对比

## 六、决策点清单（Checkpoint）

每个 Checkpoint 应产出一份 `AI_agent/experiments/<date>_<milestone>/decision.md`，记录：

| 字段 | 示例 |
|---|---|
| Checkpoint 名称 | `CP1_opus_baseline` |
| 评测数据 | `reports/summary.csv`（5 档指标 + 视觉专项）|
| 当前落在四象限哪里 | 「视觉 75% < 90%，流水线 IDF 54% < 75% → 第三象限（还在爬坡）」|
| 决策 | 「继续 P1 CoT 改造，不 Pivot」 |
| 下个 Checkpoint 预期时间 | `CP2_after_cot_refactor`，Week 5 |
| 触发条件 | zone 匹配率 ≥ 85% OR 连续两轮无提升 |

## 七、一句话行动准则

> **Opus 到不了 90% / 75% 阈值前，禁止谈 Pivot 到开源模型。没有评测数据前，禁止谈 Opus 到没到。所以真正的 next step 只有一件事：把 [plan.md](plan.md) P0 评测基线跑出来。**
